Portfolio

Context:

My portfolio page is a both a showcase of my learning outcomes and my development skills. The page itself is meant to showcase fun things I’ve been learning, incorporated into the design itself. I like developing fun projects, so for me it was important to incorporate this into my site.

Version control:

From the beginning I’ve found it important to keep a Github repo where changes are kept and so I can always have a backup when something breaks. I keep this regularly updated.

Development:

My initial portfolio design originated from learning and playing around with Javascript, challenging myself to make interesting concepts. After I've received feedback to take a few steps back and think about how my website should look and how it should function. Inspired by Star Trek's LCARS interface, I've created a design that to me is both visually appealing and functional.

JSON loader:

An important part for ease of use in developing my website is making a JSON based loader. Instead of having to manually add projects to the website, I can now just add a new project to the JSON file and it will automatically be added to the website, including the appropriate images and links. The loader also gives the URL an identifier so that the page can be bookmarked or remembered without having to reload the JSON, as it isn't loading in a seperate HTML page.

Click here to see the JSON loader.

GSAP and animation:

I had no experience with GSAP but I've wanted to make my site animated for a long time, which you can see the result of on this same page. GSAP is a great easy to use library that lets me animate my page without having to use keyframes (as often).

Three.js Globe

Another part of my portfolio is the spinning globe. This was made using Three.js. I had to find earth textures online and I learned how to apply bump maps and normal maps to a three.js sphere. I applied my basic knowledge of 3D-modeling to come up with that solution. It was a fun challenge to make and I’m happy with the result.

Webpack:

I had to find a way to easily bundle my Javascript code and libraries into one package. This was done so that I didn't have conflicts with packages and things not loading in. I found out about using Webpack and this allowed me to bundle everything I needed for Three.js into one single file. This was something that took a bit of troubleshooting to set up but it had to be done.

Easy to understand code:

Something which is important to me is that my code is easy to understand to both me and other people, that's why I always add comments to my code and use 1TBS indentation. Seen on the Github.

Reflection:

I enjoy developing fun things, I hope I’ll be able to get more opportunities for development. One thing I had issues with was the JSON loader. Making it from scratch was a nice challenge and I had thought this would make it more manageable to add in new content. However, this proved to not be the case. Adding in images inside text proved to be too much of a hassle and the entire website would look the same. Text would always be in roughly the same place, with no breaks in between. That’s why I had to make the tough decision to go back to regular HTML and CSS.

Fontys OVP

Context:

The OVP site needed a redesign to attract new students as well as improving visual clarity and reducing clutter. One of the important aspects of this project was to incorporate an easter egg.

Click the image to view the finished site.

Set up:

The website itself was made in multiple parts by the three of us. I did the initial rough layout based on the Figma file, others did the navbar, picture viewer and a few miscellaneous parts.

Tools used:

I made a Github repository in which all three of use could work in their own branch, then eventually merging it and solving conflicts. It was important to use variables for ease of use when activating the easter egg because this keeps things nice and tidy.

Easter egg:

The easter egg makes the page look like it is taking place in the Matrix. It changes the CSS root so that all elements change consistently. It also changed pictures using a system I came up with where it would point every image to a counterpart in a different folder. This saved me a lot of work and shows that it pays off to come up with something novel than finding existing code online.

Not that I didn’t also do that, because in the case of the typical Matrix style cascading text in the background, I decided it was better to find someone’s existing code. That saved me time which allowed me to focus on other parts of the site.

I also implemented the Konami code where it listens to the input from the user, also indicating the wrong key is pressed.

Reflection:

We each have different coding styles so making it work together can be a massive challenge. Thankfully by setting clear rules in a few areas, such as using one css, using the same basic variables and working in multiple branches, we managed to complete the site and successfully merge our work.

Sushootica

Context:

One of the lessons in development class was animating a div using CSS and Javascript. I took this challenge further to create Subshootica, an entire game with custom made graphics and a complete gameplay loop.

Game setup:

Instead of using a canvas, I decided on expanding the challenge in the same way as originally intended: using divs. This meant making complex Javascript code to generate and keep track of divs. As well as making them interact with the player (the mouse cursor) and other objects. The player can shoot a torpedo, that collides with enemies. Points are saved and collected and the game includes a health system as well as having powerups.

Reflection:

If I were to recreate the game now, I would definitely use a canvas, as well as setting a clear goal from the beginning. Right now, I iterated a lot based on feedback and gameplay changed because of it, but I didn’t set a goal. It was a learning process so I changed a lot of code during development which can be seen in the Git repo.
